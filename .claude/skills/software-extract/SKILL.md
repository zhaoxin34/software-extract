# software extract

用于萃取软件中的精华部分，生成提示词，提示词可用于创建新的工程。

## 什么是"精华"

萃取的"精华"是**架构级设计**，包括：
- 架构设计模式（如 Clean Architecture、分层模式）
- 核心抽象接口（如 Provider、Storage 抽象）
- Pipeline 编排模式
- 关键技术决策
- 可复用的提示词模板

**不是精华**：
- 详细的业务规则（这是 SDD Spec 的职责）
- 具体的 CLI 命令格式细节
- 每个输入/输出的格式细节

## 工作流程

### 步骤 1：获取分析目标

用自然语言获得用户的分析目标目录。如果目标是 GitHub URL，则先要 clone 项目到临时文件夹。

> **notice**: 临时文件夹用当前项目的 `temp` 目录
> **notice**: 如果是 git 项目且用户指定了分支，需要先切换到指定的分支。如果包含未提交内容导致不能切换分支，需要提示用户提交或 stash 代码。

### 步骤 2：初步分析

进入目标目录，大致了解项目的结构和意图，然后生成一个初级报告。该报告主要用于接下来生成分析任务。

**分析内容包括**：
- 项目类型（源码项目/文档项目/配置项目等）
- 技术栈
- 项目结构
- 核心功能概述

> **notice**: 这步要判断项目的价值。如果遇到项目质量特别低（比如项目下没有可以用于创建新工程的内容），则提示用户，然后退出。

### 步骤 3：创建存放目录

创建抽取结果的存放目录。

> **notice**: 抽取结果存放目录 = `{当前目录}/raw/{待萃取目标项目名称}`

### 步骤 4：保存初级报告

将上面生成的初级分析报告保存到结果目录。

> **notice**: 初级分析报告文件名 = `primary-report.md`

### 步骤 5：萃取精华

根据初级分析报告，对每个模块/子系统进行深度萃取，生成 `extracted-knowledge.md`。

**重点分析以下内容**：

1. **核心算法目录**：如 `core/`、`utils/`、`algorithms/` 等
   - 不要只写"技术工具"或"分块"
   - 要深入分析具体的算法实现（如 markdown_chunking.py 的分块策略）
   - 提取：算法名称、核心逻辑、关键特性、fallback 策略

2. **抽象接口**：如 `providers/`、`storage/`、`interfaces/`
   - 分析抽象基类的设计
   - 提取：接口方法、配置模型、扩展点

3. **Pipeline**：如 `pipelines/`
   - 分析业务流程编排
   - 提取：流程步骤、错误处理、回滚机制

4. **每个源码文件**都要看一遍：
   - 优先看有详细注释的文件
   - 提取：设计决策、降级策略、关键逻辑

**判断"精华"的标准**：
- 是否有独特的算法或策略？
- 是否有 fallback 机制？
- 是否有可复用的设计模式？

## 萃取格式模板

### 整体结构

```yaml
项目名称:
  description: 项目描述
  purpose: 存在目的/解决的问题

  模块1:
    description: 模块功能描述
    purpose: 存在目的
    responsibilities: [职责1, 职责2]

    features:
      - name: Feature名称
        description: 功能描述
        specs:
          - name: Spec名称
            description: 规格描述
            decision: 关键设计决策

    interfaces:
      - name: 接口名称
        input: 输入
        output: 输出
        usage: 使用场景

    tech_stack: [技术1, 技术2]

  模块2:
    ...

  关键设计决策:
    - 决策1: 理由
    - 决策2: 理由

  可复用的提示词模板: |
    创建类似系统...
```

### Feature/Spec 结构

```yaml
features:
  - name: Feature名称
    description: 功能描述
    specs:
      - name: Spec名称
        description: 规格描述
        implementation: 实现要点（如有源码）
        decision: 关键设计决策
```

## 萃取优先级

1. **数据模型**（如有）：表结构、字段定义、关系
2. **业务逻辑**：核心功能、流程、规则
3. **技术架构**：模块划分、依赖关系、技术选型
4. **设计决策**：为什么这样设计，有什么替代方案
5. **核心算法/策略**：重要的实现细节、算法策略（如分块策略、搜索策略等）

## 萃取维度

### 必选维度

| 维度 | 说明 |
|------|------|
| description | 模块/功能的功能描述 |
| purpose | 存在目的/解决的问题 |
| responsibilities | 核心职责列表 |

### 可选维度（根据项目特点选择）

| 维度 | 适用场景 |
|------|---------|
| features | 有明确功能划分的模块 |
| interfaces | 有抽象接口的模块 |
| tech_stack | 技术栈明确的模块 |
| patterns | 使用了设计模式的模块 |
| key_files | 关键源文件路径 |
| dependencies | 依赖的其他模块 |
| algorithms | 核心算法/策略实现（如分块策略、搜索算法等） |
| fallback_strategies | 回退策略/降级方案 |

### Feature/Spec 维度

| 维度 | 说明 |
|------|------|
| name | Feature 或 Spec 的名称 |
| description | 功能/规格描述 |
| decision | 关键设计决策 |
| implementation | 实现要点（如有源码） |

## 边界情况处理

### 无源码项目（仅有文档）

如果目标项目没有源代码，只有文档：
- 聚焦萃取：产品概念、数据模型设计、业务流程
- 从文档中提取设计意图和架构思路

### 非 Git 项目

- 直接分析本地目录结构
- 无需处理分支切换

### 二进制文件或大文件

- 跳过二进制文件
- 大文件（如 >10MB）可以采样或跳过
- 优先分析源码文件（.py, .js, .ts, .go, .java 等）

### 萃取失败

- 如果某个模块分析失败，记录错误并继续
- 最终报告中标注分析失败的模块
- 提示用户哪些部分需要手动补充
